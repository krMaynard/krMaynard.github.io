Prompts Are Refactoring Targets: Logging an AI Workflow to Find What to Automate

June 19, 2026
AI/LLM Engineering Workflow

Anyone who works with an AI coding agent every day eventually notices they keep typing the same things. "Commit this, push it, open a PR." "Address the review comments." "Fix the failing CI." It feels repetitive — but a feeling is a weak basis for deciding what to automate. A recent engineering post from LayerX (by Yohei) replaces the feeling with data: log every prompt you send the agent, analyze the log on a schedule, and let the patterns that actually recur tell you what to build. It's a small idea with a sharp framing, and I think it's right.

The phrase that stuck with me is that prompts become refactoring targets. We already treat code as something to observe, measure, and reshape into reusable pieces; the argument here is that your instructions to an AI deserve the same treatment. Saying the same thing to an agent a dozen times is duplication — and duplication in code is a smell precisely because it's a signal you've found a unit worth extracting.

How it works

The mechanics are deliberately modest. A UserPromptSubmit hook appends each prompt to a local prompts.jsonl — one JSON line with a timestamp, a session id, and the text — skipping trivially short inputs and keeping the file owner-readable only. Logs are masked, kept locally, and deleted after seven days. (That last part isn't an afterthought: a prompt log is a record of everything you've asked, so the privacy guardrails are load-bearing, not optional.)

Two detection paths run on top of the log. One works inside a live session, noticing in real time when the same operation comes up again. The other is a scheduled daily job that reads the past week of prompts and groups them by intent. Running both catches more than either alone: the real-time path sees bursts within a conversation, the batch path sees the same request scattered across days. When a pattern crosses a low threshold — three or more occurrences — it posts to Slack, with the pattern summary and a ready-made prompt for building the corresponding Skill or command attached.

Two design choices make this more than a logging gimmick. First, the daily analysis uses an LLM to group prompts semantically rather than clustering embeddings — so "make a PR," "commit, push, and open a PR," and "commit the fix and get a PR up" all collapse into one pattern, because they share an intent regardless of wording. That's exactly the kind of fuzzy, judgment-laden grouping language models are good at and rigid similarity metrics are bad at. Second, the automation stops at proposing. It never creates or deletes a Skill on its own; a human approves each one. That keeps the system from quietly breeding a thicket of half-useful rules — the failure mode that makes most "self-improving" automation a liability.

It doesn't stop at new Skills

The part I found most interesting is where the system turned on itself. If people keep invoking an existing Skill and then immediately tacking on more instructions — "also update the Notion page," "also post to Slack," "you didn't finish" — that trailing chatter is evidence the Skill is underspecified: unclear completion criteria, a missing follow-up step, a description the model can't reliably discover. So the same log that surfaces new automations also surfaces broken ones. Skills, like code, aren't write-once; you refactor them as you watch them get used.

The reported results, as of this month: 29 distinct patterns identified across 691 occurrences, 22 of them (76%) converted into Skills or commands, eliminating 652 repeated prompts. The commit-push-PR cycle alone showed up 94 times before it became a single /ship-style command; handling review comments, 92 times; chasing red CI, 76. Those aren't headline-grabbing numbers, and that's the point — they're the boring, high-frequency toil that's invisible until you count it.

What I take from it

This rhymes with how I already work with Claude Code. When I had it security-review my research API, the wins came from running a finite checklist exhaustively; when I use LLMs in compliance PM work, they come on structured tasks with verifiable outputs. Prompt-log analysis is the same shape pointed at my own habits: it's structured information extraction over a corpus I generate by working, with a human verifying every proposed change. The author frames it as Scrum's empiricism — transparency, inspection, adaptation — and the comparison lands. Logging is transparency; the daily review is inspection; converting a pattern to a Skill is adaptation. What AI changes is the size of the loop: the unit between "I keep doing this by hand" and "this is now a command" shrinks to days.

The judgment stays where it belongs. The log tells you what you repeat; it can't tell you whether a repetition is worth abstracting, whether a Skill encodes the right defaults, or whether a workflow should exist at all. Those are the same calls that separate a good refactor from a premature one. But that's the appeal of the framing: it doesn't promise to replace the judgment — it just makes sure you're spending it on the things you actually do, not the ones you only think you do. I'm going to start logging mine.
