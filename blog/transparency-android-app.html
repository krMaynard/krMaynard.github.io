Putting the Transparency Dashboard in Your Pocket

July 15, 2026
Kotlin Multiplatform Compose Android Engineering

The transparency query API has had a web dashboard for a while. What it didn't have was a home-screen icon. So the data explorer is now a native app, it runs on Android and on the desktop, from a single Kotlin Multiplatform codebase, and it talks to the exact same public API the website does.

You can download the Android build and sideload it today.

Why an app at all

Most of this data is something you check, not something you sit and study, a removal trend, how one platform's numbers compare to another's, what a given country's requests look like this period. That's a phone interaction, not a desktop one. A dashboard that assumes a wide screen and a mouse makes the quick look harder than it should be. The app is built the other way around: phone first, and the wide layouts are what expands from there, rather than the reverse.

One codebase, two platforms

Everything meaningful lives in one shared source set: the DTOs, the Ktor client that calls /api/overview and /api/explore, the repository that builds each query, and every screen and chart. Compose Multiplatform renders that same UI on Android and in a desktop window. Only two things are per-platform, the entry point (an Activity on Android, a window on the desktop) and the HTTP engine, wired through Kotlin's expect/actual so Android gets OkHttp and the desktop gets CIO. There's no second implementation of the app to keep in sync, because there's no second implementation.

The same discipline that keeps the API honest carries into the client: it never sends SQL. A query is a structured ExploreRequest, a table name plus typed filters, group-bys, and aggregates, that the server validates and compiles. The app builds those requests; it doesn't build strings. Adding a new dataset view is a repository method, not a hand-written query.

Mobile-first, and it means it

The layout measures its own width and adapts. Below a breakpoint it's a phone: bottom navigation, a single column, and an Explore flow that goes list → detail the way you'd expect on a small screen. Above it, the same screens expand into a navigation rail and a master-detail view. The screens are stateless. A single view-model exposes immutable UI state as a StateFlow, so the same composables render on a phone, in a desktop window, and in the screenshot process that produced the frames above.

Charts are drawn on the Compose canvas rather than pulled from an image service, so they're crisp at any density and honest about what they show, bands stay bands, rates never get summed, and a period with no data reads as no data instead of a zero.

The Android details worth doing

A native app should feel native, which is a pile of small things: an adaptive launcher icon with a monochrome layer for themed icons, launcher shortcuts that jump straight to the removals view or the dataset browser, predictive-back so the system's back gesture previews where you're headed, and per-app language settings that surface in Android's own list, the app is already translated into eight languages, so it honors whichever one you pick there. The release build runs through R8 with resource shrinking, which is how a full Compose app lands around 1.5 MB.

Get it

The Android APK is a direct download, enable installs from your browser and open it. It's the same data, the same query model, and the same caveats as the web dashboard; it just lives on your home screen now. The desktop build runs from the same codebase if you'd rather have it in a window.
